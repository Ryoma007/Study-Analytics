# 持久化存储选型：SQLite + better-sqlite3

后端要持久化已结算的 ActivitySession，并承接统计聚合计算。场景特征：单用户自托管、Docker 部署、数据量极小（个人学习记录最多几万条）、低写入（停止计时写一条）、需要按日/周/月/年聚合。

决定使用 SQLite + `better-sqlite3` 同步驱动。理由：数据随容器卷走、备份即复制文件、SQL 聚合天然适配统计需求、单用户微秒级查询不会造成可感知阻塞。

**项目级例外**：本项目使用 `better-sqlite3` 同步 API，构成对用户全局规则"nodejs 项目不要使用 fs 的 Sync 同步 api"的明确例外。例外仅限 SQLite 数据访问层；其它文件 I/O 仍遵守禁同步规则。SQLite 同步查询微秒级、仅短时持锁，不构成全局规则所针对的"阻塞事件循环导致程序无法继续执行"风险。后续不得以"遵循全局规则"为由改回异步驱动——这是有意为之的权衡。
