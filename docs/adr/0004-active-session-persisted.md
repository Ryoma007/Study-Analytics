# ActiveSession 持久化为 SQLite 单行表

引入后端心跳架构后，ActiveSession（运行中会话）的存储形态有两个选择：纯内存态（重启即丢，靠前端 410 后自动恢复）或持久化。

决定将 ActiveSession 落 SQLite 单行表 `active_session`（始终最多一行），后端不维护内存副本，所有心跳与结算直接读写 DB。

理由：
1. 单一事实源，不存在内存态与持久态双写同步的复杂度，重启/崩溃天然恢复。
2. 懒结算正确性依赖持久化的 `last_heartbeat_at`——若纯内存，重启后无从判断一个"其实早已超时"的会话，会误判为"还在运行"。
3. better-sqlite3 同步读写微秒级，心跳 15s 一次 UPDATE、单用户单行表，性能无忧，不需要内存缓存。
4. 比前端"410 后自动恢复"方案简单：后者要前端处理 410 + 重发 start + 保持原 startTime，跨时钟跨重启语义模糊；持久化让后端始终权威，前端逻辑简单。

代价：每次心跳一次 DB 写，但单用户场景可忽略。
